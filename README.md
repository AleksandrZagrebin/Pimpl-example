# User Pimpl Example

Пример реализации паттерна **Pimpl (Pointer to Implementation)** на современном C++ с использованием `std::unique_ptr`, move‑семантики и RAII.

Проект создан как учебный пример для отработки ключевых концепций C++, важных при разработке системного и высокопроизводительного кода.

## Особенности реализации

- **Pimpl‑паттерн**: скрытие деталей реализации класса `User` через указатель на внутреннюю структуру `Impl`.
- **RAII**: управление временем жизни `Impl` делегировано `std::unique_ptr`.
- **Move‑семантика**: разрешены перемещение и присваивание с гарантией `noexcept`; копирование запрещено.
- **Разделение интерфейса и реализации**: публичный API вынесен в `user.hpp`, детали реализации — в приватный модуль.
- **Чистота зависимостей**: клиентский код (`main.cpp`) не зависит от внутренних деталей `Impl`.

Такой подход уменьшает время перекомпиляции в больших проектах и стабилизирует ABI.

---
### Почему важно поддерживать стабильный ABI

Это критично для библиотек, системного ПО и ядра:

- **Обновление без пересборки клиентов.** При стабильном ABI можно заменить библиотеку на новую версию, и уже собранные приложения продолжат работать. Если ABI сломан, все клиенты придётся пересобирать.
- **Снижение рисков в продакшене.** В больших системах пересборка всех компонентов — дорого и рискованно. Стабильный ABI позволяет обновлять компоненты точечно.
- **Совместимость бинарных дистрибутивов.** Библиотеки часто распространяются в виде `.so`/`.dll`. Пользователи ожидают, что минорные обновления не сломают совместимость.
- **Предсказуемость зависимостей.** Для системного программиста это значит меньше неожиданных проблем при интеграции.

### Как Pimpl помогает сохранять ABI

В этом проекте паттерн Pimpl напрямую решает проблему стабильности ABI:

- В публичном заголовке `user.hpp` `Impl` объявлен только как неполный тип (`struct Impl;`).
- Размер класса `User` всегда равен размеру одного `std::unique_ptr<Impl>`. Даже если добавить в `Impl` новые поля или изменишь их типы, размер `User` не изменится.
- Клиенты, использующие `user.hpp`, не зависят от внутренней структуры `Impl`. Поэтому любые изменения в реализации не требуют перекомпиляции клиентского кода.

## Структура проекта

- `user.hpp` — публичный интерфейс класса `User`.
- `user_impl.hpp` — приватное определение структуры `Impl` (используется только в реализации).
- `user.cpp` — реализация методов класса `User`.
- `main.cpp` — пример использования класса `User`.

## Сборка и запуск
g++ -std=c++17 -O2 main.cpp user.cpp -o app
./app
